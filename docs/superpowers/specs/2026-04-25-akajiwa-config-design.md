# Akajiwa — Claude Code Config Design

**Date:** 2026-04-25
**Author:** Reky Bongso
**Status:** Approved

\---

## Overview

A personal Claude Code config system that adapts over time to the taste and standards of a solo PM + product engineer + UX designer. The config is layered (global to project), task-driven via slash commands, and self-upgrading through a memory loop that captures in-session corrections and promotes recurring patterns into permanent guidance.

**Design goal:** Reduce AI slop over time by accumulating real taste corrections into the config so Claude stops producing the things the user consistently fixes.

\---

## Architecture

### Approach

Layered config (global → project) + task-specific slash commands + self-upgrading memory loop. No role personas. Commands are task-shaped, not role-shaped.

### Repository structure

```

├── skills/
```

### How layers interact

```
Global CLAUDE.md           -> always active, all projects
  |
Project .claude/CLAUDE.md  -> active only in that project folder
  |
Slash command skill        -> activated on demand
  |
Memory (shared + project)  -> informs all of the above over time
```

## Global CLAUDE.md

### Identity and context

* Senior Solo PM + Senior product engineer + Senior UI/UX designer
* Builds web products: Next.js, React, Express, Node, MongoDB
* Standards apply across code, design, and product work

### Code standards

* Strict TypeScript everywhere — no `any`, no untyped variables
* Functional programming preferred, OOP only for external connectors
* Pure functions only — never mutate inputs or global state
* No default parameter values — all params explicit
* No silent error handling — always raise with context
* No fallbacks unless explicitly asked
* All imports at top of file
* Functions under 30 lines — if longer, it is doing too much
* No comments unless the WHY is non-obvious
* Explicit return types on all functions
* Proper type definitions for all complex data structures

### Stack defaults

* Frontend: Next.js / React, strict TypeScript
* Backend: Express + Node, strict TypeScript
* Database: MongoDB + Mongoose
* Testing: Playwright (E2E), Vitest (unit)
* Package manager: pnpm

### SonarQube quality gate (new code)

* New bugs and vulnerabilities: 0
* Security hotspots reviewed: 100%
* Coverage on new code: >= 80%
* Duplicated lines on new code: <= 3%

### SonarQube critical rules

* Re-exports (S4328): always `export { x } from './y'` — never import then export
* Cognitive complexity: functions must stay <= 15
* DRY strings (S1192): no repeated string literals — extract to constants
* MUI v7: use `slotProps.input` not `inputProps`

### Anti-slop standards

* No generic placeholder copy ("No items yet", "Click here to get started")
* No transactional CTAs ("Submit", "Create Account") — prefer action-forward tone
* No AI-looking UI patterns — flat hierarchies, generous whitespace, intentional typography
* API error responses must be machine-readable codes, not natural language strings
* Hitboxes in game or interactive projects reduced to \~80% of visual size for feel
* Playtest physics constants before hardcoding them

### Token efficiency (always on)

* Use `rtk` prefix for all terminal commands
* Keep responses focused — no trailing summaries, no re-explaining what was just done
* Small focused files — if a file grows large, flag it as doing too much
* Prefer reading source code over guessing library behavior

### Model reminders

Heavy commands check the current model and remind the user to switch to Opus if not already on it. Light commands run fine on Sonnet or Haiku with no reminder.

## Slash Commands

### Model guide

|Command|Recommended model|Reminder|
|-|-|-|
|`/xplan`|Opus|Yes|
|`/xdesign`|Opus|Yes|
|`/xreview`|Opus|Yes|
|`/xqa`|Sonnet|No|
|`/xvibe-fe`|Haiku (simple) / Sonnet (complex)|No — /xplan determines|
|`/xvibe-be`|Haiku (simple) / Sonnet (complex)|No — /xplan determines|
|`/xvibe-design`|Haiku (simple) / Sonnet (complex)|No — /xplan determines|
|`/xinit`|Sonnet|No|
|`/xretro`|Sonnet|No|
|`/xreflect`|Sonnet|No|
|`/update-memory`|Haiku|No|
|`/get-memory`|Haiku|No|

### Command definitions

**`/xplan`** — Plan a feature or change before any code is written.

* Checks for existing `.claude/CLAUDE.md` — creates it if missing (same as `/xinit`)
* Reads `memory/features-index.md` to surface relevant existing features
* Determines scope from the feature description: FE only, BE only, or full stack
* Assesses difficulty per scope: simple or complex
* Asks focused questions scoped to what is actually needed (data model, API routes, component breakdown)
* Generates `docs/plans/\[feature]-context.md` and `docs/plans/\[feature]-plan.md`
* Appends feature entry to `memory/features-index.md` when plan artifacts are generated
* Recommends which vibe command and model to use at the end of the plan:
  - FE only, simple → `/xvibe-fe` on Haiku
  - FE only, complex → `/xvibe-fe` on Sonnet
  - BE only, simple → `/xvibe-be` on Haiku
  - BE only, complex → `/xvibe-be` on Sonnet
  - Full stack → lists FE and BE separately with their own difficulty and model recommendation
* Opus recommended — shows reminder if on Sonnet or Haiku

**Difficulty criteria:**

| Simple | Complex |
|---|---|
| Color, font, spacing changes | Business logic, multi-step workflows |
| Basic CRUD endpoints | Authentication, authorization flows |
| Config changes | State machines, data transformations |
| Copy edits | Complex queries, aggregations |
| Renaming, restructuring | Performance optimization |
| Standard form components | Algorithm implementation |

**`/xdesign`** — UI/UX design brief and direction.

* Covers: visual direction, component breakdown, copy tone, interaction patterns, responsive behavior
* Reads `memory/features-index.md` to check for existing components to reuse
* Generates `docs/plans/\[feature]-design.md`
* Opus recommended — shows reminder if on Sonnet or Haiku

**`/xreview`** — Fresh-context code or design review.

* Reads the diff with no prior session context
* Flags issues against quality standards, SonarQube rules, and anti-slop standards
* Opus recommended — shows reminder if on Sonnet or Haiku

**`/xvibe-fe`** — Fast-iteration mode for frontend.

* Minimal explanation, no scaffolding comments, small focused outputs
* Context: components, hooks, UI, styling, Next.js patterns
* Use Haiku for simple tasks (colors, fonts, layout tweaks, standard forms)
* Use Sonnet for complex tasks (state management, complex component logic, data-heavy views)
* /xplan recommends the model at the end of the plan — follow that recommendation
* Co-op skills (optional — skill works fully without them):
  - `frontend-dev-guidelines` — invoke for React/TypeScript component work (primary)
  - `frontend-design-pro` — invoke when the component needs distinctive visual treatment
  - `debugging` — invoke when component behavior is unexpected
* If a co-op skill is not installed, proceed with built-in guidance silently — do not mention it to the user

**`/xvibe-be`** — Fast-iteration mode for backend.

* Minimal explanation, no scaffolding comments, small focused outputs
* Context: Express routes, Mongoose models, middleware, services
* Use Haiku for simple tasks (basic CRUD, config changes, renaming)
* Use Sonnet for complex tasks (business logic, auth flows, complex queries, data transformations)
* /xplan recommends the model at the end of the plan — follow that recommendation
* Co-op skills (optional — skill works fully without them):
  - `databases` — invoke for MongoDB schema design, Mongoose queries, aggregations, indexing (primary)
  - `backend-development` — invoke for complex service layer, API architecture, middleware design
  - `debugging` — invoke when a route or query produces unexpected results
* If a co-op skill is not installed, proceed with built-in guidance silently — do not mention it to the user

**`/xvibe-design`** — Fast-iteration mode for UI/UX design.

* Minimal explanation, no scaffolding comments, small focused outputs
* Context: layout, visuals, copy, interaction patterns
* Use Haiku for simple tasks (copy edits, color changes, spacing)
* Use Sonnet for complex tasks (full layout design, interaction patterns, design systems)
* /xplan recommends the model at the end of the plan — follow that recommendation
* Co-op skills (optional — skill works fully without them):
  - `ui-ux-pro-max` — invoke for full design direction and UI/UX intelligence (primary)
  - `aesthetic` — invoke when the interface needs opinionated visual beauty
  - `web-design-guidelines` — invoke after design to validate against UI guidelines
* If a co-op skill is not installed, proceed with built-in guidance silently — do not mention it to the user

**`/xqa`** — Generate a Playwright test plan for a flow.

* Covers: happy path, edge cases, error states, persistence
* Generates test file scaffold alongside the plan

**`/xinit`** — Set up project config for an existing project without planning.

* Creates `.claude/CLAUDE.md` from a template based on answers to setup questions
* Creates `.claude/memory/memory.md`
* Reads existing codebase to detect conventions
* Use when joining an existing project with no specific feature to plan yet
* For new projects or when ready to plan immediately, use `/xplan` instead

**`/xretro`** — Session retrospective.

* Scans conversation for corrections, discoveries, patterns, anti-patterns
* Classifies each learning and routes it to the right memory layer
* Project-specific learnings → `.claude/memory/memory.md`
* Universal candidates → flagged for `/xreflect`
* Also runs automatically via Stop hook at session end

**`/xreflect`** — Promote recurring patterns to global config.

* Scans both memory layers for patterns appearing 3 or more times
* Proposes promotions to global `CLAUDE.md` or relevant skill
* On approval: edits the file, commits, pushes to config repo
* Removes duplicate entries from memory after promotion

**`/update-memory`** — Push project memory to project repo.

* `git add .claude/memory/memory.md`
* Commits with auto-generated message from latest retro entries
* Pushes to project repo so teammates receive the learnings

**`/get-memory`** — Pull latest shared memory and config updates.

* `git pull` on config repo
* Re-symlinks skills if anything was added or changed
* Reports what changed (memory entries, skill updates, CLAUDE.md changes)

## Self-Upgrading System

### The learning loop

```
User corrects Claude output mid-session
  |
Session ends -> Stop hook fires /xretro automatically
  |
/xretro scans conversation, classifies learnings:
  CORRECTION   -> something Claude got wrong that the user fixed
  DISCOVERY    -> something unexpected that worked well
  PATTERN      -> approach that consistently produces good results
  ANTI-PATTERN -> approach that consistently fails
  |
Routes to the right memory layer:
  Project-specific  -> <project>/.claude/memory/memory.md
  Universal candidate -> flagged, queued for /xreflect
  |
When pattern appears 3x -> /xreflect proposes a config change
  |
User approves -> Claude edits CLAUDE.md or skill, commits, pushes to config repo
  |
Teammates run /get-memory -> change propagates
```

### Stop hook config (settings.json)

```json
{
  "hooks": {
    "Stop": \[
      {
        "matcher": "",
        "hooks": \[
          {
            "type": "command",
            "command": "claude /xretro"
          }
        ]
      }
    ]
  }
}
```

### Retro rules

* One line per learning, with date
* Non-obvious only — skip typos and trivial renames
* Never duplicate an existing entry
* Always classify: CORRECTION, DISCOVERY, PATTERN, or ANTI-PATTERN

### Reflect rules

* Pattern must appear 3 or more times across different sessions
* Always propose before changing — user approves every promotion
* After promotion: commits and pushes to config repo automatically
* Removes duplicate entries from memory to keep it clean

## Memory Architecture

### Three layers

|Layer|File|Loaded|Scope|
|-|-|-|-|
|Shared standards|`memory/memory.md` (config repo)|Always|All projects, all users|
|Features index|`memory/features-index.md` (config repo)|On demand|Cross-project feature awareness|
|Project learnings|`.claude/memory/memory.md` (project repo)|Always in project|That project only|

### Entry format

```md
## 2026-04-25

- CORRECTION: API error strings changed to machine-readable codes (e.g. "account\_already\_registered").
  Prefer error codes at API boundary, not natural language.

- PATTERN: Playtest physics constants before hardcoding — math-derived values
  consistently adjusted after first playtest.

- ANTI-PATTERN: Generic empty state copy ("No tasks yet!") consistently rewritten.
  Always write grounded, specific copy from the start.
```

### Features index format

```md
## Profile Feature
- Project: todo-app
- Repo: github.com/yourname/todo-app
- Implemented: 2026-03-10
- What: User profile with avatar, display name, bio
- Exposes:
  - API: GET /api/users/:id
  - Component: <UserProfile userId={id} />
  - Hook: useProfile()
- Data: User { id, displayName, avatarUrl, bio }
```

### Features index behavior

* Read only by `/xplan` and `/xdesign` — not loaded in every session
* `/xplan` appends new entry when a plan is finalized
* `/xretro` updates existing entry if API or data model changed during session
* Changes committed and pushed to config repo automatically

### Sync commands

```
/update-memory    -> commit + push project memory to project repo
/get-memory       -> git pull config repo (shared memory + updated skills)
```

```md
- \[Core entities and their relationships]
```

**How it stays updated:**

## Installer

### What it installs

|Task|install.sh|install.ps1|
|-|-|-|
|Global CLAUDE.md -> \~/.claude/CLAUDE.md|Yes|Yes|
|Skills -> \~/.claude/skills/ (symlinked)|Yes|Yes|
|Stop hook -> \~/.claude/settings.json (merged)|Yes|Yes|
|RTK PreToolUse hook -> settings.json|Yes|Yes|
|Memory directory + memory.md|Yes|Yes|
|RTK binary (auto, OS + arch aware)|Yes|Yes|
|RTK PATH setup|Yes|Yes|

### RTK installation behavior

**install.sh (Mac/Linux):**

1. Detect OS (darwin / linux) and architecture (x64 / arm64)
2. Fetch latest RTK version from GitHub releases API
3. Download correct binary for detected OS + arch
4. Move to `/usr/local/bin/rtk` or `\~/.local/bin/` if no sudo
5. chmod +x
6. Verify: `rtk --version`

**install.ps1 (Windows):**

1. Detect architecture (x64 / arm64)
2. Fetch latest RTK version from GitHub releases API
3. Download `rtk.exe` for detected arch
4. Move to `C:\\tools\\rtk\\rtk.exe`
5. Add `C:\\tools\\rtk` to user PATH (no admin required)
6. Verify: `rtk --version`

**If RTK already installed:** skip download, verify version, continue.

**If download fails:**

```
Could not download RTK (no internet or GitHub unreachable).
Install manually: https://github.com/rtk-ai/rtk/releases
Everything else was installed successfully.
```

### Co-op skill detection

At the end of installation, the installer checks for optional co-op skills in `~/.claude/skills/` and reports missing ones:

```
Optional co-op skills not found (install for enhanced results):
  frontend-dev-guidelines  ->  used by /xvibe-fe
  ui-ux-pro-max            ->  used by /xvibe-design
  databases                ->  used by /xvibe-be

Your config works fully without them.
```

If all co-op skills are present: "All optional co-op skills found."

### Updating on an existing machine

```
/get-memory
```

Pulls latest shared memory, updated skills, and any CLAUDE.md changes from the config repo.

## How-to Guide (README.md)

### Installation

**Prerequisites:** Claude Code, Git, internet connection.

**Mac/Linux:**

```bash
git clone https://github.com/yourname/akajiwa-config
cd akajiwa-config
chmod +x install.sh
./install.sh
```

**Windows:**

```powershell
git clone https://github.com/yourname/akajiwa-config
cd akajiwa-config
./install.ps1
```

**Updating on an existing machine:** run `/get-memory`

### Commands

**Heavy — switch to Opus first. Commands will remind you if you forget.**

|Command|What it does|
|-|-|
|`/xplan`|Plan a feature before building. Generates plan artifacts in docs/plans/|
|`/xdesign`|UI/UX design brief and direction. Generates design artifact in docs/plans/|
|`/xreview`|Fresh-context code or design review|

**Light — Sonnet or Haiku.**

|Command|What it does|
|-|-|
|`/xvibe-fe`|Fast-iteration mode for frontend — components, hooks, UI, styling|
|`/xvibe-be`|Fast-iteration mode for backend — routes, models, middleware|
|`/xvibe-design`|Fast-iteration mode for design — layout, visuals, copy|
|`/xqa`|Generate a Playwright test plan for a flow|
|`/xinit`|Set up project config without planning a feature|
|`/xretro`|Session retrospective (also runs automatically on session end)|
|`/xreflect`|Promote recurring patterns to global config|
|`/update-memory`|Push project memory to project repo|
|`/get-memory`|Pull latest shared memory and config updates|

### Workflow

**Any project (new or existing):**

1. Open Claude Code in your project folder
2. Run `/xplan` and describe what you want to build

   * First time in this project: setup runs automatically, no `/xinit` needed
3. Build. Use `/xvibe-fe`, `/xvibe-be`, or `/xvibe-design` — `/xplan` recommends which one
4. Run `/xqa` before finishing a flow
5. Run `/xreview` before committing
6. End the session — `/xretro` fires automatically via Stop hook

**Setting up an existing project without a specific feature to plan:**

* Run `/xinit` once. Then use `/xplan` for all future sessions.

**Keeping config up to date:**

* Corrections you make during sessions are captured by `/xretro` automatically
* Patterns that repeat 3 or more times: `/xreflect` proposes a config update
* You approve — pushed to config repo
* Other machines or teammates run `/get-memory` to receive the update

**Model guide:**

* Opus: `/xplan`, `/xdesign`, `/xreview`
* Sonnet: `/xqa`, `/xinit`, `/xretro`, `/xreflect`
* Haiku or Sonnet (determined by /xplan difficulty assessment): `/xvibe-fe`, `/xvibe-be`, `/xvibe-design`
* Haiku: `/update-memory`, `/get-memory`

## Rules and constraints

* No emojis anywhere in config files, command outputs, or generated artifacts
* Memory entries are non-obvious learnings only — no trivial corrections
* Config repo contains only universal, reusable items — no project-specific artifacts
* Artifact files (plans, PRDs, charters) live in the project repo under docs/
* `/xreflect` never edits config without explicit user approval
* Installer merges into existing settings — never overwrites
* Installer never creates fake credentials or modifies .env files
